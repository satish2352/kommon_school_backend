1. What is Kommon School?

Kommon School Backend is the brain behind a student enrollment + online payment platform. It handles the entire journey: from a student filling out an "Enroll Now" form on the website, to paying through Razorpay, to being provisioned into a third-party learning platform called Sumago.
Imagine a coaching class that takes online admissions. A student visits the website, fills a form, pays the fees online, and gets login credentials to start learning. This backend is the kitchen behind that restaurant — students see a clean menu (frontend), but behind the scenes, this backend cooks the meal: it stores the form data, takes the money safely, calls Razorpay, waits for confirmation, and then tells another system (Sumago) to give the student their textbooks. It also keeps shopkeeper-side tools (admin panel) so the team can manage admissions, payments, follow-ups, and reports.

What problems does it solve?

  • Take online payments safely using Razorpay (no money should be lost, ever).
  • Handle thousands of simultaneous enrollments without creating duplicate records or duplicate charges.
  • Provision students automatically on the Sumago learning platform after successful payment.
  • Retry automatically if any step fails (network outage, Sumago down, etc.) — no manual babysitting.
  • Give admins control via dashboards: see enrollments, payments, follow-ups, reports, retry failed syncs.
  • Track everything for audit — every privileged action is logged.

2. The Big Picture — Architecture

The system isn't just one Express server. It's a small ecosystem with multiple moving parts that talk to each other through PostgreSQL and Redis.

Think of the backend like a busy post office.
  • The Express server is the front counter where customers (students, admins) come and submit forms.
  • PostgreSQL is the big filing cabinet where all permanent records live.
  • Redis is the sticky-notes board — fast, temporary, used to remember "this job needs doing" or "I'm currently working on this, don't touch it".
  • BullMQ workers are the postmen who go out and deliver letters (call Sumago's API) in the background. They retry if a delivery fails.
  • Cron jobs are the night-shift clerks who clean up stale paperwork at fixed schedules.

Why this design?

  • Decouple slow work from fast responses. When a student pays, the user-facing response returns in milliseconds. The slow part (calling Sumago, retrying on failure) happens in a worker after the response is sent.
  • Survive Sumago outages. If Sumago is down, jobs queue up in Redis and retry later instead of failing the student's enrollment.
  • Scale horizontally. You can run multiple Express+Worker instances on different servers. Redis distributed locks make sure cron jobs only run on ONE instance at a time.

5. How the Server Starts (boot order)

When you run npm start, here's what happens, in order:

sequenceDiagram participant SH as Shell participant SRV as server.js participant APP as app.js participant ENV as config/env.js participant PG as PostgreSQL participant RD as Redis participant Q as BullMQ Worker participant C as Cron Scheduler SH->>SRV: node src/server.js SRV->>APP: require('./app') APP->>ENV: validate environment variables ENV-->>APP: ok (or process.exit if missing) APP->>APP: register middleware + routes SRV->>SRV: app.listen(PORT) SRV->>Q: startQueues() → Worker connects to Redis SRV->>C: startJobs() → schedule 6 cron jobs SRV->>PG: getPrismaClient() (eager connection) PG-->>SRV: connection pool ready Note over SRV: Server ready to accept traffic SRV-->>SH: log "server_started"

Graceful shutdown

When SIGTERM (Ctrl+C, PM2 reload, Docker stop) arrives, the server doesn't just die. It:

  1. Stops accepting new HTTP requests.
  2. Stops scheduling new cron ticks (in-flight ticks finish).
  3. Drains BullMQ workers — lets active jobs complete (up to 15s).
  4. Disconnects Prisma cleanly.
  5. Disconnects Redis.
  6. Exits cleanly (or forces exit after 15s timeout).

Why both candidate_type and external_sync_status?

Think of an enrollment as a parcel.
  • candidate_type = WHERE the parcel came from (the website front door or the admin office).
  • EnrollmentStatus = whether the parcel was paid for (the postage stamp).
  • external_sync_status = whether the parcel was actually delivered to Sumago (the destination).
These are independent because the parcel can be paid for but stuck in transit to Sumago.

8. The Golden Path — Student Enrollment Journey

This is the most important flow in the entire system. Memorize it.

sequenceDiagram autonumber participant U as Student Browser participant API as Express API participant DB as PostgreSQL participant RZ as Razorpay participant Q as BullMQ Queue participant W as Worker participant SU as Sumago Note over U,API: Step 1 — Submit "Enroll Now" form U->>API: POST /api/v1/enrollments {name, email, phone} API->>DB: BEGIN tx; SELECT FOR UPDATE on email DB-->>API: existing row OR none alt new enrollment API->>DB: INSERT enrollment (status=submitted) else resume API->>DB: UPDATE enrollment with new form data end API->>DB: COMMIT API-->>U: {id, enrollmentCode, created/resumed} Note over U,API: Step 2 — Pick a plan U->>API: PATCH /api/v1/enrollments/:id/plan {planPricingId} API->>DB: validate plan ACTIVE; cancel stale orders; SET plan_pricing_id API-->>U: ok Note over U,RZ: Step 3 — Create Razorpay order U->>API: POST /api/v1/enrollments/:id/payment-order API->>DB: SELECT FOR UPDATE on enrollment API->>DB: check no existing success payment API->>RZ: orders.create({amount, currency}) RZ-->>API: {orderId} API->>DB: INSERT Payment (status=initiated) API->>DB: UPDATE enrollment.status = payment_pending; COMMIT API-->>U: {paymentId, razorpayOrderId, keyId, amount} Note over U,RZ: Step 4 — Pay with Razorpay U->>RZ: Open Razorpay checkout, enter card/UPI RZ-->>U: success + {paymentId, signature} Note over U,API: Step 5 — Verify payment U->>API: POST /api/v1/enrollments/:id/payment-verify API->>API: HMAC(orderId|paymentId, secret) == signature ? API->>DB: BEGIN tx (Serializable); SELECT FOR UPDATE on Payment DB-->>API: latest row alt already success API-->>U: {status:success, alreadySettled:true} else first to settle API->>DB: UPDATE Payment.status=success API->>DB: UPDATE Enrollment.status=paid; external_sync_status=PENDING API->>DB: COMMIT API->>Q: enqueueExternalApiSync(jobId=enrollmentId) API-->>U: {status:success} end Note over Q,SU: Step 6 — Background sync (async) Q->>W: Pick up job W->>DB: load enrollment + payment W->>SU: POST {firstName, email, phone, plan, transactionId...} alt 2xx success SU-->>W: 200 W->>DB: UPDATE external_sync_status=SUCCESS; log SUCCESS else 5xx / timeout SU-->>W: error W->>DB: log RETRYING; throw Note over Q: BullMQ retries with backoff 30s→60s→120s→240s→480s else after 5 retries W->>DB: external_sync_status=DEAD_LETTER; auto-create followup end Note over RZ,API: Step 7 — Razorpay webhook (parallel safety net) RZ->>API: POST /api/v1/webhooks/razorpay payment.captured API->>API: verify HMAC signature on raw body API->>DB: idempotency check (event_id unique) API->>DB: settlePayment() — same race-protected logic API-->>RZ: 200 ok
Read the diagram top to bottom like a story:
  1. Student submits form → row created in DB.
  2. Student picks plan → row updated with planPricingId.
  3. Student clicks "Pay" → backend asks Razorpay for an order, stores it, returns the orderId.
  4. Razorpay shows the payment popup. Student pays.
  5. Razorpay sends payment success info back to the frontend. Frontend tells the backend "verify this".
  6. Backend checks the signature (proof Razorpay sent it), marks the payment as paid, and queues a job to tell Sumago.
  7. Worker picks up the job, calls Sumago. If it fails, retry up to 5 times. If still failing, mark dead-letter and create a marketing follow-up.
  8. Meanwhile, Razorpay also sends a webhook to us directly — this is a safety net in case the frontend never called verify (e.g., browser crashed). The same race-protected code runs, whichever arrives first wins.

9. Authentication & JWT Tokens

src/modules/auth/ + src/config/jwt.js + src/middleware/auth.middleware.js

How login works

sequenceDiagram autonumber participant U as User participant API as API participant DB as DB U->>API: POST /auth/login {email, password} API->>DB: find user by email DB-->>API: user row (or null) API->>API: bcrypt.compare(password, hash) — always runs (timing safety) alt password ok API->>API: generate familyId (UUID) API->>API: generate tokenId/jti (UUID) API->>API: sign accessToken (15 min) — claims {sub, email, role} API->>API: sign refreshToken (7 days) — claims {sub, fam, jti} API->>DB: INSERT refresh_tokens (token_hash = SHA256(rawToken), family_id) API->>DB: INSERT audit_logs (action=auth.login) API-->>U: {accessToken, refreshToken, user} else password wrong API-->>U: 401 INVALID_CREDENTIALS end

Why two tokens?

  • Access token (15 min): Sent on every request as Authorization: Bearer .... Short-lived so a stolen one expires fast.
  • Refresh token (7 days): Stored more carefully (httpOnly cookie or secure storage). Used only to get new access tokens.

Token rotation & reuse detection

Imagine your refresh token is a movie ticket stub. Every time you use it to get a new access token, the box office punches a hole in it (marks it revoked) and gives you a brand new stub. If anyone tries to use the punched stub again, the box office knows it's stolen and cancels every ticket in your booking family — you have to log in again from scratch. This is called "refresh token rotation with reuse detection". It catches stolen tokens.

Family tracking

Every login creates a new family_id. Every refresh keeps the same family_id but creates a new jti. If a token in a family is ever reused after revocation, all tokens in that family are revoked.

10. RBAC — Roles & Permissions

Two layers of access control are stacked:

  1. Coarse role check — authorize(['admin', 'superadmin']) from middleware/rbac.middleware.js
  2. Granular permission check — hasPermission('payments:retry') using permissions / role_permissions tables

Default role × permission matrix

Superadmin always passes. The middleware short-circuits on role === 'superadmin' without hitting the DB. Permission rows are still seeded for completeness, but they're not consulted.

In-process cache

The hasPermission middleware caches per-role permission lookups for 60 seconds. Means: changing permissions in the DB propagates within a minute. To evict immediately, restart the process.

5-minute dedupe window

Before the transaction, the service checks: did this (email, phone, status=submitted) arrive in the last 5 minutes? If yes, return the existing row as-is. Reason: double-click protection — pressing the submit button twice shouldn't create two records.

  • KS — Kommon School prefix (constant)
  • 2605 — current UTC year-month (YY + MM)
  • A1B2C3 — 6 random hex chars

Imagine the student keeps refreshing the payment page. Each refresh hits this endpoint. Without protection, we'd create a fresh Razorpay order every time — confusing for everyone. The "reusable" logic finds an existing initiated order with the right amount and hands it back. Existing stale orders (wrong amount) get cancelled. At most one live Razorpay order per enrollment at any time.

Verification flow (race-protected settlement)

flowchart TD A[POST /enrollments/:id/payment-verify

Why SELECT FOR UPDATE?

Two paths can settle the same payment: (1) the student's browser calling /verify, (2) Razorpay's server sending a webhook to us. They could arrive at the exact same millisecond. Without locking, both might:
  1. Read the Payment row (both see status='initiated').
  2. Both update to status='success'.
  3. Both enqueue a Sumago sync → two emails to the student!
SELECT FOR UPDATE says "the first one to read this row locks it; the second one waits at the door". When the second one finally gets in, it sees status='success' already and returns "alreadySettled" without doing anything. This is the most important pattern in the codebase.

13. Razorpay Webhook Handling

src/modules/razorpay/webhook.controller.js + src/middleware/webhookRaw.middleware.js

Razorpay independently sends events to POST /api/v1/webhooks/razorpay when payments happen. We trust these less than the verify endpoint (because the signature is the only proof), so verification is strict.

sequenceDiagram autonumber participant RZ as Razorpay participant MW as webhookRaw middleware participant CTL as webhook.controller participant DB as DB RZ->>MW: POST /webhooks/razorpay {body} + x-razorpay-signature MW->>MW: Capture raw bytes BEFORE express.json() MW->>CTL: pass through req.rawBody CTL->>CTL: HMAC-SHA256(rawBody, webhook_secret) == signature? alt invalid CTL-->>RZ: 401 (but Razorpay will retry — that's fine) else valid CTL->>DB: INSERT webhook_events ON CONFLICT (event_id) DO NOTHING alt duplicate event CTL-->>RZ: 200 (idempotent — already processed) else new event CTL->>CTL: dispatch by event_type alt payment.captured / order.paid CTL->>DB: settlePayment() — same race-protected logic else payment.failed CTL->>DB: mark payment failed if not already settled end CTL->>DB: UPDATE webhook_events SET processed=true CTL-->>RZ: 200 end end

Why raw body?

Express normally parses JSON and discards the original bytes. HMAC verification needs the exact bytes Razorpay signed — even a single whitespace change breaks the signature. So webhookRaw middleware captures the raw buffer before express.json() can mangle it. Notice in app.js that the webhook route is registered BEFORE app.use(express.json()).

Idempotency by event_id

Razorpay retries failed webhooks. The webhook_events.event_id column is UNIQUE — a second arrival of the same event_id fails the INSERT, signaling "already processed". We respond 200 immediately without re-processing.

Always 200

Even if internal processing fails, we return 200 to Razorpay. Otherwise they retry. We record the failure in webhook_events and a cron (webhook_retry) replays it later.

Circuit breaker (opossum)

If Sumago is clearly down (50% of calls failing), instead of hammering it 100 times in a minute we "open the circuit" — every call fails instantly for 30 seconds. Then we try one to see if it's back. This protects Sumago from a thundering-herd of retries AND saves our worker's time.

Idempotency at the queue level

Imagine you double-click "verify payment". Both clicks enqueue a sync job. With jobId = enrollmentId, BullMQ sees the second one and discards it silently (while the first is still waiting or active). Without this, you'd risk calling Sumago twice for the same student.

16. Cron Jobs & Distributed Locks

src/jobs/ + src/utils/distributedLock.js

Why distributed locks?

Say you scale to 3 server instances (instance-A, instance-B, instance-C). Each one has its own cron scheduler. At 2:00 AM, all three try to "expire stale enrollments". Without coordination, you'd do the same work three times. Distributed locks ensure only ONE instance does it. The other two see "lock is held" and skip cleanly.

The lock mechanism

Uses Redis SET key value NX EX ttl (atomic set-if-not-exists with expiry). The "value" is a random UUID per acquire. To release, a Lua script checks the value matches before DELETE — so if the lock expired and someone else grabbed it, we don't accidentally release theirs.

The six scheduled jobs

JobCronLock TTLWhat it does
payment_reconciliation */5 * * * * 4.5 min For payments pending > 10 min, ask Razorpay the real status. Settle captured ones. Expire uncaptured after 30 min.
external_api_retry */2 * * * * 1.75 min Reschedule external_api_log rows stuck at retrying (e.g., after Redis restart wiped in-memory BullMQ jobs).
webhook_retry */3 * * * * 2.75 min Replay webhook_events rows with processed=false and attempts < 5.
enrollment_cleanup 0 2 * * * 10 min Expire submitted enrollments older than 24h.
refresh_token_cleanup 0 3 * * * 10 min Hard-delete revoked/expired refresh tokens older than 30 days.

20. Error Handling Strategy

Five categories

  1. Validation errors — Joi rejects → 422 with field details
  2. Authorization errors — Missing/invalid token → 401; insufficient role/permission → 403
  3. Business rule errors — Thrown as ApiError with a code → maps to a specific HTTP status
  4. Prisma errors — P2002 unique violation, P2025 not found, etc. → friendly codes
  5. Unexpected errors — Anything else → 500 generic + full log with stack + traceId

The graceful degradation principle

  • If Redis is down → enqueue fails → fall back to inline HTTP call to Sumago (fire-and-forget).
  • If Sumago is down → BullMQ retries → eventually dead-letter → marketing follow-up auto-created.
  • If Razorpay webhook signature fails → respond 401 but always 200 on processed-but-errored (let Razorpay retry).
  • If audit log write fails → log and continue (never block a business action).

TermMeaning
